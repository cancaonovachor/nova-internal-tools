# Notion Webhook → Discord通知連携 基本設計書

## 1. 概要

本システムは、NotionのWebhookイベントをトリガとして、対象イベントをDiscordへ通知する仕組みを提供する。

高頻度更新に耐えるため、Webhook受信と通知処理を分離し、非同期処理基盤を用いたスケーラブルな構成とする。

---

## 2. 背景・課題

NotionのWebhookは以下の特性を持つ：

* ページ更新が高頻度で発火する
* 同一ページに対する連続更新が多い
* 同一イベントが複数回配送される可能性（at-least-once）

そのため、以下の課題が存在する：

* Discord通知のスパム化
* リソース負荷の増大
* 重複通知
* 一時的なスパイクによる処理遅延

---

## 3. 設計方針

### 3.1 基本方針

* Webhook受信処理は最小化する
* 通知処理は非同期化する
* 冪等性を担保する
* イベントの間引き・集約を行う

---

### 3.2 アーキテクチャ

```
Notion Webhook
   ↓ HTTPS
Cloud Run (ingress)
   ↓ Cloud Tasks
Cloud Run (worker)
   ↓
Discord
```

---

## 4. コンポーネント設計

### 4.1 Cloud Run (ingress)

#### 役割

* Notion Webhookの受信
* 署名検証
* イベントフィルタリング
* タスク投入
* 即時レスポンス

#### 処理内容

1. 署名検証
2. イベント種別判定
3. 必要情報抽出
4. Cloud Tasksへ投入
5. HTTP 2xxレスポンス返却

#### 非機能要件

| 項目            | 値              |
| ------------- | -------------- |
| CPU           | 0.08〜0.25 vCPU |
| Memory        | 256MiB         |
| Concurrency   | 50〜200         |
| Timeout       | 5〜10秒          |
| Min instances | 0              |

---

### 4.2 Cloud Tasks

#### 役割

* 非同期処理のキューイング
* リトライ制御
* スパイク吸収

#### 設計ポイント

* task名に一意キーを付与（重複排除）
* リトライポリシー設定

---

### 4.3 Cloud Run (worker)

#### 役割

* イベント処理
* 重複排除
* デバウンス
* Discord通知

#### 処理内容

1. 冪等性チェック
2. デバウンス判定
3. Notion API追加取得（必要時）
4. Discord通知
5. エラーハンドリング

#### 非機能要件

| 項目          | 値             |
| ----------- | ------------- |
| CPU         | 0.25〜0.5 vCPU |
| Memory      | 256〜512MiB    |
| Concurrency | 1〜20          |
| Timeout     | 30〜60秒        |

---

## 5. イベント制御設計

### 5.1 イベントフィルタリング

対象イベント例：

* `comment.created`
* `page.created`

制御対象：

* `page.content_updated`（制限あり）

---

### 5.2 デバウンス

同一ページに対する短時間更新を抑制する。

#### 方法

* `page_id`単位で最終更新時刻を保持
* 一定時間内（例：60秒）の再通知を抑制

---

### 5.3 冪等性

#### 方法

以下のキーで重複排除：

```
event_type + page_id + time_bucket
```

または

```
event_id（存在する場合）
```

---

## 6. データストア（任意）

デバウンス・冪等性のために使用：

* Firestore
* Redis（Memorystore）

### 保存情報

| 項目               | 内容     |
| ---------------- | ------ |
| page_id          | ページID  |
| last_notified_at | 最終通知時刻 |
| event_hash       | 重複判定用  |

## 7. Discord通知設計

### 7.1 通知内容

* 操作ユーザー
* イベント種別
* ページタイトル
* ページURL

### 7.2 レート制限対策

* 429時のリトライ
* 通知間隔制御
* 集約通知

---

## 8. スケーリング設計

### 8.1 スパイク対応

* ingressは軽量処理のみ
* Cloud Tasksでバッファリング

### 8.2 同時実行制御

* ingress: 高め
* worker: 低め（Discord制約考慮）

---

## 9. 障害対応

### 9.1 想定障害

| 障害            | 対応     |
| ------------- | ------ |
| Discord API失敗 | リトライ   |
| Notion API失敗  | 再取得    |
| タスク失敗         | 自動リトライ |
| 重複通知          | 冪等性制御  |

---

## 12. 実装優先順位

1. ingress + worker + Cloud Tasks
2. イベントフィルタリング
3. 冪等性
4. デバウンス
5. 通知整形
6. 監視・ログ

---

## 13. 補足（設計の意図）

本設計では「リソースを増やす」のではなく、

* 不要イベントを捨てる
* 処理を遅延させる
* 同じイベントをまとめる

ことで負荷を制御している。
